# 1. 📂 プロジェクト全体概要（全員共有用）

## プロジェクトの目的と概要

本プロジェクトでは、複数人で遊べるオンラインお絵描きクイズゲーム（通称：お絵描きの森）をJavaのソケット通信を用いて制作します。

この制作の最大の裏テーマは「クライアント・サーバー間のネットワーク通信を学ぶこと」です。そのため、フロントエンドとバックエンドで分業するのではなく、機能ごとに縦割り（A・B・C）で担当を分け、全員が「画面UI」から「サーバー側でのデータ処理」まで一貫して開発を経験できる体制をとります。

## ゲームの基本ルール

1. 誰か1人が「部屋（ルーム）」を作成し、他の人がそこに参加する。
2. 全員揃ったらゲーム開始。システムがランダムに「絵を描く人」と「当てる人」に分ける。
3. 描く人には「お題（例：りんご）」が提示され、キャンバスに絵を描く。
4. 当てる人はチャットで回答を送信。早く正解した人ほど高得点。
5. 制限時間が来るか、全員が正解したらその回は終了。
6. 全員が1回ずつ「描く人」を担当したら1ラウンド終了（任意ラウンド繰り返し）。
7. 最終的なスコアを表示して終了。

## 開発体制と役割分担

衝突（コンフリクト）を防ぐため、以下の4つの役割に完全に分割して開発を進めます。

* **担当A：ルーム管理**（部屋の作成・参加・待機）
* **担当B：お絵描き**（マウス座標の取得・描画・リアルタイム同期）
* **担当C：ゲーム進行・チャット**（お題の管理・正誤判定・スコア計算）
* **担当D（基盤）：通信土管の作成**（サーバー・クライアント間の接続維持とデータ振り分け）※作成済み

## 開発の絶対ルール：通信コマンド（Protocol）

全員が1つの通信回線（ソケット）を共有するため、勝手にデータを送るとバグの原因になります。
データを送受信する際は、必ず `common/Protocol.java` に「コマンド名」を定義し、「コマンド:データ内容」という形式（例：`CHAT_SEND:りんご`）で通信を行ってください。新しい通信が必要になったら、まずはチームで合意をとってProtocolに追加しましょう。

これまでのやり取りと、共有していただいた画像に基づいた**最新のディレクトリ構造**と、**各ファイルが具体的に何をしているのかの解説**をまとめました。

このリストが、チーム全員にとっての「全体マップ」になります。

### 📄 各ファイルの役割解説

#### 【共通領域】全員が参照するルールブック

* **`common/Protocol.java`**
* **役割：** 通信で使う「合言葉（コマンド）」を定数として定義するファイルです。
* **なにしてるか：** 例えば「線を引いたよ」というデータを送るときに、`"DRAW"` という文字列を使うのか `"LINE"` を使うのか、チーム内で表記揺れが起きないようにここで `public static final String DRAW_LINE = "DRAW_LINE";` のように定義し、全員がこれを参照します。



#### 【サーバー側】基盤担当（D）の領域

* **`server/GameServer.java`**
* **役割：** サーバーの「受付係」です。
* **なにしてるか：** ポート番号（8080など）を指定して起動し、新しいプレイヤー（クライアント）が接続してくるのをずっと待ち構えます。誰かが接続してきたら、その人専用の `ClientHandler` を新しく作って対応を任せます。


* **`server/ClientHandler.java`**
* **役割：** 各プレイヤー専属の「電話交換手」です。
* **なにしてるか：** 繋がったプレイヤーから送られてくる文字列（例：`ROOM_CREATE:ライオン部屋`）を常時読み取ります。そして、`Protocol.java` の合言葉をもとに「これは部屋作成の依頼だから Aさん（RoomManager）に渡そう」「これはお絵描きデータだから Bさん（DrawManager）に渡そう」と、**`switch` 文を使って各Managerへ処理を丸投げ（振り分け）します。**



#### 【サーバー側】A・B・Cの領域（処理の実行部）

* **`server/room/RoomManager.java` (担当A)**
* **役割：** 部屋の管理者です。
* **なにしてるか：** `ClientHandler` から「部屋を作って」「部屋に入りたい」という依頼を受け取り、現在ある部屋のリストや、誰がどの部屋にいるかという情報をサーバーのメモリ上で管理・更新します。


* **`server/draw/DrawManager.java` (担当B)**
* **役割：** お絵描きデータの中継地点です。
* **なにしてるか：** 誰かが絵を描いたときの座標データを `ClientHandler` から受け取り、同じ部屋にいる「他の全員の `ClientHandler`」を探して、その座標データをそのまま一斉送信（ブロードキャスト）します。


* **`server/game/GameManager.java` (担当C)**
* **役割：** ゲームの審判であり、進行役です。
* **なにしてるか：** お題を選んで各プレイヤーに配ったり、送られてきたチャット内容がお題と一致しているか（正解か）を判定したりします。また、制限時間やスコアの計算もここで行います。



#### 【クライアント側】基盤担当（D）の領域

* **`client/GameClient.java`**
* **役割：** プレイヤーのPC（クライアント）側の通信窓口です。
* **なにしてるか：** サーバーのIPアドレスとポートを指定して接続（Socket通信）を行います。また、「いつサーバーからデータが送られてきてもいいように、常に耳をすませておく裏方のプログラム（受信スレッド）」を走らせ、データが届いたらクライアント側の各Controllerへ振り分けます。



#### 【クライアント側】A・B・Cの領域（各自がこれから作るファイル）

* **`client/room/` (担当A)**
* **なにしてるか：** プレイヤーが部屋名を入力する画面（UI）や、その入力内容を `ROOM_CREATE` コマンドとしてサーバーに送信する処理（Controller）をここに作っていきます。


* **`client/draw/` (担当B)**
* **なにしてるか：** マウスで線を描けるキャンバス画面（UI）と、マウスが動いた座標を取得してサーバーに送信したり、逆にサーバーから送られてきた他人の線を描画したりする処理（Controller）をここに作っていきます。


* **`client/game/` (担当C)**
* **なにしてるか：** チャットを入力する画面や残り時間を表示する画面（UI）と、入力したチャット内容をサーバーに送信したり、サーバーから「正解！」という結果を受け取って画面にスコアを表示したりする処理（Controller）をここに作っていきます。

# 2. 📝 担当A：ルーム管理チーム 仕様書

**ミッション：**
プレイヤーがゲームを始めるための「ロビー（待合室）」を作ることです。誰がどの部屋にいるのかを管理し、BさんやCさんがゲームを始められる土台を整えます。

## 実装する機能

1. **部屋の作成**：プレイヤーが任意の部屋名を入力して部屋を作れるようにする。
2. **部屋への参加**：すでに作成されている部屋の一覧を表示し、選んで参加できるようにする。
3. **待機画面の表示**：現在同じ部屋に誰がいるのかリスト表示し、「ゲーム開始」ボタンを配置する（開始処理自体はCが担当）。

## 編集する主なファイル

* `client/room/RoomPanel.java` (画面：部屋作成・一覧表示UI)
* `client/room/RoomController.java` (送信：サーバーへのリクエスト送信)
* `server/room/RoomManager.java` (受信：サーバー側での部屋リスト・メンバー管理)

## 通信のイメージ

* **送信**：`ROOM_CREATE:ライオン部屋` （部屋「ライオン部屋」を作って！）
* **受信**：`ROOM_CREATED_NOTIFY:ライオン部屋,成功` （サーバーからの作成完了通知）
* **送信**：`ROOM_JOIN:ライオン部屋,ユーザー名` （その部屋に入れて！）

## 開発のヒント

まずは見た目（UI）をシンプルに作り、ボタンを押したときに `System.out.println` で正しく文字列が生成されるか確認するところから始めるとスムーズです。

# 3. 🎨 担当B：お絵描きチーム 仕様書

**ミッション：**
このゲームの心臓部である「キャンバス」を作ることです。一人が描いたマウスの軌跡をデータに変換し、他の全員の画面にリアルタイムで魔法のように表示させます。

## 実装する機能

1. **お絵描きキャンバス**：マウスのドラッグ操作を検知し、画面に線を引く。
2. **色の変更・消しゴム**：ペンの色を変えたり、キャンバスをクリアしたりする機能。
3. **座標データの送受信**：「どこからどこまで線を引いたか」の座標データをサーバーに送り、他人が描いた座標データを受け取って自分の画面にも描画する。

## 編集する主なファイル

* `client/draw/DrawPanel.java` (画面：キャンバスUI、マウスイベント取得)
* `client/draw/DrawController.java` (送信：座標データの文字列化と送信)
* `server/draw/DrawManager.java` (受信：受け取った座標を同じ部屋の他メンバーに転送)

## 通信のイメージ

* **送信**：`DRAW_LINE:150,200,152,205,BLACK` （X1,Y1からX2,Y2へ黒い線を引いたよ！）
* **受信**：`DRAW_LINE:150,200,152,205,BLACK` （サーバー経由で他の人の線データが届く）

## 開発のヒント

マウスが1ミリ動くたびに通信するとデータがパンクします。最初はカクカクでもいいので、座標を文字列（カンマ区切りなど）に変換して送る・受け取って元に戻す（シリアライズ・デシリアライズ）処理に集中しましょう。

# 4. 🎮 担当C：ゲーム進行・チャットチーム 仕様書

**ミッション：**
ゲームの「ルールと審判」を作ることです。いつ始まり、誰が絵を描き、誰が正解し、いつ終わるのかという状態（ステート）を管理し、プレイヤーに熱狂をもたらします。

## 実装する機能

1. **チャット機能**：プレイヤーがテキストを入力して送信し、全員の画面に表示させる。
2. **お題の選出と役割振り分け**：「描く人」にはお題をこっそり教え、「当てる人」には隠す。
3. **正誤判定とスコア管理**：送られてきたチャットがお題と完全一致するかチェック。正解ならスコアを加算し、制限時間やターンの管理を行う。

## 編集する主なファイル

* `client/game/ChatPanel.java` (画面：チャット入力欄、ログ表示、スコア・残り時間表示)
* `client/game/GameController.java` (送信：チャット発言の送信)
* `server/game/GameManager.java` (受信：正誤判定、タイマー処理、全員への結果通知)

## 通信のイメージ

* **サーバーから**：`GAME_ROUND_START:DRAWER,りんご` （あなたは描く人！お題はりんご！）
* **送信**：`CHAT_SEND:みかん` （みかんだと思う！）
* **サーバーから**：`GAME_JUDGE:USER1,不正解` （USER1の回答はハズレです）
* **サーバーから**：`SCORE_UPDATE:USER1=50,USER2=10` （現在の得点状況）

## 開発のヒント

一番ロジックが複雑な担当です。まずは純粋な「チャットアプリ（入力した文字が全員に届く）」を作ることを第一目標にしてください。それができたら、サーバー側で「入力された文字 == お題」を判定するIF文を追加していくと分かりやすいです。

# 5. 📌 現在の実装状況（機能・動作仕様）

初期の仕様書作成後に実装が進み、以下のルールが動くようになっています。混乱防止のため実際の挙動をまとめます。

## ロビー〜ゲーム開始までの流れ

1. 部屋を作成/参加すると、まず**ロビー画面（部屋一覧・待機中メンバー）に留まる**。お絵描き画面へは自動遷移しない。
2. ロビー画面の「ゲーム開始」ボタンは「自分の準備完了」を送信するボタン。押すと `GAME_START:部屋名` が送られる。
3. サーバー（`GameManager`）は部屋にいる**全員が「ゲーム開始」を押すまで**ラウンドを開始しない。誰か1人が押した時点では `GAME_READY:準備人数,合計人数,準備済みユーザー名` が全員に配信され、ロビー画面に「準備完了: n / m」として表示される。
4. 全員分が揃うと、サーバーが `G_R_START`（ラウンド開始）を配信し、そのタイミングで初めてクライアントはお絵描き画面へ切り替わる。

## Drawer / Guesser の役割制限

- ラウンドごとに1人が「Drawer（描く人）」、それ以外は「Guesser（当てる人）」としてランダムに割り当てられる。
- **描画キャンバスを操作できるのはDrawerだけ。** Guesser側では `DrawPanel` へのマウス操作・色変更・Clearボタンが無効化され、キャンバスにはうっすら網掛け表示が入る（`DrawController.setDrawingEnabled(false)` により制御）。ただし、Guesser側にもキャンバス自体は常に表示され、Drawerが描いた線はリアルタイムで見える（描けないだけで見えないわけではない）。
- お題（Theme）は `G_R_START` のデータの中で**Drawerにのみ**含まれて配信される。Guesser側には空文字が届くため、画面には `Theme: (guessers can't see this)` と表示される。これはバグではなく「当てる人にお題を見せない」という仕様どおりの挙動。

## 通信・ゲーム進行の安定性

### クライアント・サーバー間通信の安定化

- `client/GameClient.java`の接続処理に5秒のタイムアウトを設定し、サーバーへ到達できない場合に早くエラーを確認できるようにした。
- クライアントとサーバーの入出力をUTF-8へ統一し、Windows・WSL・macOS間でも日本語の部屋名やユーザー名を正しく送受信できるようにした。
- 接続状態を確認してから送信し、切断処理を一箇所へ集約した。受信処理は識別しやすい名前付きDaemonスレッドで実行する。
- 日本語の部屋名「日本語テスト部屋」とユーザー名「アリス」「ボブ」を使ったソケット統合テストで、文字化けせず部屋を作成・参加できることを確認済み。

### 描画コマンドの権限・入力検証

- `server/draw/DrawManager.java`で、描画・全消去の送信者が対象ルームの正式なメンバーかを確認する。
- ゲーム中は`server/game/GameManager.java`の`canDraw()`で現在のDrawerかも確認し、Guesserや部屋外のクライアントから届いた描画コマンドを破棄する。
- 描画座標がすべて整数であること、色と線幅が許可された値であることをサーバー側で検証する。
- 全消去は`DrawController.requestClear()`からサーバーを経由し、同じルームの全クライアントへ同期される。

### ゲーム中のルーム状態・切断処理・タイマーの安定化

- ゲーム開始後の途中参加、ゲーム中の別ルームへの移動・再参加を拒否し、準備人数・役割・スコアの整合性を維持する。
- メンバー変更時は`RoomManager`から`GameManager`へ通知し、準備状態とゲーム状態を同期する。
- ゲーム中に参加者が2人未満になった場合はゲームを安全に中断し、残った参加者へ終了を通知する。
- 準備状態の削除範囲を対象ルーム内に限定し、切断時もMapを安全に更新する。
- 正解人数とDrawerボーナスの集計を修正し、タイマーを固定遅延方式へ変更して、処理遅延後に時刻通知が連続することを防いだ。

## 表示・操作体験

### ゲーム開始・役割交代直後のTheme／Time表示

- `GamePanel`のステータス欄を2行、`ChatPanel`を3行へ整理し、狭い画面でもThemeとTimeが表示領域外へ折り返されないようにした。
- ラベル更新後に`revalidate()`と`repaint()`を実行し、ウィンドウサイズを変更しなくても表示が更新されるようにした。
- `GameController`はTheme・Timeを含むラウンド情報を各パネルへ反映してからゲーム画面へ切り替える。初期Timeは`GamePanel`と`ChatPanel`の両方へ設定される。
- 幅560pxのゲーム領域を含む`ClientDispatchTest`で、開始直後・役割交代直後ともThemeとTimeが表示範囲内に収まることを確認済み。

### ゲーム終了後の準備完了表示リセット

- 通常終了時と切断による中断時にサーバー側の準備状態を削除し、`GAME_READY_UPDATE`で現在の`0 / 人数`を同じルームの全員へ通知する。
- 2人ゲームの通常終了後は「準備完了: 0 / 2」、1人が切断した中断後は残ったクライアントに「準備完了: 0 / 1」と表示される。

### フリーボード型の描画UI

- 描画画面をヘッダー、左側ツールドック、中央キャンバス、フッターで構成し、キャンバスを画面の主役として大きく表示する。
- ツールドックからペン、消しゴム、黒・赤・青・緑・黄・紫の6色、細・中・太の3段階の線幅、全消去を選択できる。
- `P`でペン、`E`で消しゴム、`1`〜`3`で線幅を選べる。現在のツールは背景色と枠線でも判別できる。
- `DrawPanel`は800×600の論理キャンバスを縦横比を保って拡大・縮小し、画面上のマウス座標を論理座標へ変換して送信する。カーソルには現在の線幅プレビューを表示する。
- Guesserには「回答者は絵を見て答えよう」と表示し、描画ツールを無効化する。

### 線幅を含む描画同期

- `DrawController`と`Protocol`の描画データに線幅を追加し、他クライアントでも同じ太さで再現できるようにした。
- 従来形式との互換性を保つため、線幅は省略可能な値として扱う。
- `DrawManager`で線幅の数値・許可範囲を検証し、許可色に`PURPLE`を追加した。
- 色・線幅の異なる線と全消去の同期、およびDrawer以外からの描画拒否を統合テストで確認済み。

### ロビーから最終結果までのUI／UX統一

- `client/UiTheme.java`に、森をイメージした配色、共通フォント、ボタン、パネル、入力欄のスタイルを集約した。
- ロビー、描画、回答チャット、スコア、最終結果の情報の優先順位と余白を整理し、同じ配色・言葉・操作ルールで進行できるようにした。
- DrawerとGuesserには、それぞれの役割で必要な情報だけを表示する。
- 主要部品にAccessible Name、説明、ツールチップ、キーボード操作、色以外の状態表示を追加した。

### ゲーム状態を伝えるフィードバックエフェクト

- `client/FeedbackEffect.java`を追加し、レイアウトを動かさずに重ねて表示できる共通エフェクトを実装した。
- 正解時は成功色のフラッシュ、紙吹雪、「正解！ +得点」を約1.4秒表示する。不正解・入力エラーは赤枠、残り10秒は警告、得点更新はスコア領域の強調、最終結果は紙吹雪で伝える。
- ラウンド終了時は正解と「次のラウンドを準備中」を中央へ表示し、次の状態までの進行をバーで示す。
- `-Ddrawingforest.reduceMotion=true`または`DRAWING_FOREST_MOTION=reduce|off`でアニメーションを停止し、文字による状態表示だけを残せる。

### ラウンド・最終結果への遷移タイミング統一

- 通常ラウンドは`GAME_ROUND_END`受信後、全プレイヤーが共通して1.15秒待ってからラウンド終了表示へ移る。
- 最終ラウンドは`GAME_END`受信後、全プレイヤーが共通して1.5秒待ってから最終結果画面へ移る。正解者は正解エフェクト、それ以外には「ゲーム終了・最終結果を集計中」を表示する。
- 軽減モーション設定でも待機時間は変えず、プレイヤー間の遷移タイミングを揃える。
- 次ラウンド開始、ゲーム終了、新しいゲームの初期化時には保留中のSwing Timerを停止し、古い画面遷移が後から実行されることを防ぐ。

動作確認手順そのものは `RUNNING.md` を参照してください。

# 6. 🧠 設計思想・スコアリング・想定動作の詳細

チームで実装を引き継ぐ人向けに、コードを読まないと分からない仕様・判断理由をまとめます。

## 6.1 スコアリング仕様

**Guesser（当てる人）の得点：**
```
得点 = 100 + 残り秒数 × 10
```
- ラウンド開始からの経過時間をもとに`server/game/GameManager.java`の`calculatePoints()`で計算。早く正解するほど高得点、最低保証100点。
- 1人が同じお題で複数回加点されることはない（`correctUsers`で1人1回のみに制限。2回目以降は`ALREADY_CORRECT`を返すだけで加点しない）。
- Drawer自身が送信したメッセージは正誤判定の対象外（ただのチャットとして扱われる）。

**Drawer（描く人）の得点：**
- Drawerは自分では正解判定を受けないが、**そのラウンドでGuesser全員が獲得した得点の合計 ÷ 正解者数（平均、整数除算）**をラウンド終了時にボーナス加点される（`awardDrawerBonusLocked()`）。
- 誰も正解できなかったラウンドは、Drawerへの加点はない（0で割ることになるため）。
- 設計意図：早く・分かりやすい絵を描くほど正解者が増えて平均点も上がりやすくなるよう、Drawer側にも「上手く描くモチベーション」を持たせるため。

**ラウンドの終わり方（`finishRoundLocked`のreason）：**
| reason | 意味 |
|---|---|
| `all_correct` | Guesser全員が正解した |
| `time_up` | 制限時間(60秒)が経過した |
| `drawer_left` | Drawerが退室・切断した |

## 6.2 ラウンド進行・ゲーム終了の仕組み

- ゲーム開始時、参加者全員の名前を`drawingOrder`としてシャッフルし、順番にDrawerを担当していく（`RoomGame`コンストラクタ）。
- 1人がDrawerを1回ずつ担当し終えたら（`drawerIndex >= drawingOrder.size()`）、そのタイミングでゲーム終了(`GAME_END`)。何ラウンド行うかは自動的に「参加人数と同じ回数」になる。
- ゲーム開始後の途中参加、別ルームへの移動・再参加は`RoomManager`で拒否する。メンバー変更は`GameManager`へ通知され、ゲーム中に参加者が2人未満になると残った参加者へ終了を通知して安全に中断する。
- ラウンド間は3秒のインターバルを置いてから次のラウンドを自動開始する（`finishRoundLocked`内の`Timer`）。時刻通知は固定遅延方式で実行し、処理遅延後の連続通知を防ぐ。
- 制限時間は`ROUND_TIME_SECONDS = 60`（1分固定）。`GAME_TIME_UPDATE`で毎秒配信され、`ChatPanel`/`GamePanel`双方の「Time: 秒数」ラベルに表示（残り10秒以下で赤色に変化）。
- 通常ラウンドは`GAME_ROUND_END`受信後1.15秒、最終ラウンドは`GAME_END`受信後1.5秒を全プレイヤー共通の表示時間として確保する。軽減モーション設定でも待機時間は変えず、画面遷移のタイミングを揃える。
- 次ラウンド開始、ゲーム終了、新しいゲームの初期化時は保留中のSwing Timerを停止し、前の状態の画面遷移が後から実行されないようにする。

## 6.3 ゲーム終了後の画面遷移

- `GAME_END`受信時、正解者には正解エフェクト、それ以外には「ゲーム終了・最終結果を集計中」を表示してから、結果画面(`client/game/ResultPanel.java`)へ移る。
- 結果画面はスコアの高い順にランキング表示し、1〜3位を金・銀・銅で強調する。
- 「終了」ボタンを押したときだけロビー画面（部屋一覧）へ戻り、部屋一覧を再取得する。
- 通常終了時と切断による中断時は準備状態を削除し、`GAME_READY_UPDATE`で「準備完了: 0 / 現在人数」へ更新する。前回ゲームの準備人数は次のロビーへ持ち越さない。

## 6.4 ユーザー名の一意性について

- `server/game/GameManager.java`の得点・準備状況・ラウンド管理は、いずれも**ユーザー名の文字列**をキーにしている（`ClientHandler`オブジェクトそのものではない）。
- そのため、同じ部屋で同名のユーザーが複数人いると、サーバー側では別人として区別できず、得点やラウンド管理が壊れる。
- これを防ぐため、`server/room/RoomManager.java`の`ensureUniqueName()`で、同じ部屋内の名前重複を検知した場合は自動的に連番を付けて一意化している（例：2人とも`Player`のままなら、後から参加した方は自動的に`Player2`になる）。UIの初期値が両方とも`"Player"`のままでも安全に動作する設計。

## 6.5 お絵描き機能の設計

- 線データは800×600の論理キャンバス上の座標として`DRAW_DATA:部屋ID,X1,Y1,X2,Y2,色,線幅`の形式で送る。線幅を省略した従来形式も受信できる。
- `DrawPanel`は論理キャンバスの縦横比を保ったまま表示領域へ拡大・縮小し、マウス座標と論理座標を相互変換する。そのため、参加者のウィンドウサイズが異なっても同じ位置・色・線幅で描画を再現できる。
- サーバー(`DrawManager`)は送信者のルーム所属とDrawer権限を確認し、座標・色・線幅が許可範囲内の場合だけ同室のクライアントへ中継する。
- Clear操作は`DRAW_CLEAR`として同室の他クライアントにも同期される。また、新しいラウンドが始まるたびに全員のキャンバスが自動的にクリアされる（前のラウンドの絵が残らないように、ラウンド開始(`G_R_START`)受信時に`DrawController.clearForNewRound()`を呼ぶ）。

## 6.6 UIの配色・フォント方針

- `client/UiTheme.java`に配色（深緑を基調、Drawer=オレンジ、Guesser=青、正解=緑、不正解=赤、金・銀・銅）とフォントサイズの基準を集約している。新しく画面を追加する場合はここから参照して統一感を保つこと。
- `UiTheme.installGlobalDefaults()`は`RoomLauncher.main()`の最初に一度だけ呼ぶことで、Swingの全コンポーネントの基準フォントサイズを底上げする。個別の画面でフォントサイズを指定しなくても、この呼び出しだけで全体が読みやすくなる。
- ロビーから最終結果まで情報の優先順位、余白、部品の見た目を統一し、主要部品にはAccessible Name、説明、ツールチップ、キーボード操作、色以外の状態表示を設定している。
- `FeedbackEffect`は正解時のフラッシュ・紙吹雪・得点表示、不正解時の赤枠、残り10秒の警告、得点更新の強調、ラウンド終了時の進行表示を、レイアウトを動かさずに重ねて描画する。
- `-Ddrawingforest.reduceMotion=true`または環境変数`DRAWING_FOREST_MOTION=reduce|off`を指定するとアニメーションを停止し、文字による状態表示だけを残す。

## 6.7 通信・入力検証の設計

- `GameClient`の接続タイムアウトは5秒。接続状態を確認してから送信し、切断処理を一箇所へ集約している。受信処理は名前付きDaemonスレッドで実行する。
- `GameClient`と`ClientHandler`の入出力はUTF-8に統一し、OSが異なる環境間でも日本語の部屋名・ユーザー名を扱える。
- `DrawManager`は描画・全消去の送信者が対象ルームの正式なメンバーであることを確認し、ゲーム中は`GameManager.canDraw()`で現在のDrawerかも検証する。
- 描画座標4個の整数形式、色、線幅を許可リスト・範囲で検証し、不正なコマンドは他クライアントへ転送しない。

## 6.8 自動テストと実行環境

- `tests/ProtocolIntegrationTest.java`は実TCPソケットと`ClientHandler`を使い、日本語通信、ルーム・ゲーム進行、描画権限、採点、終了・切断処理を一連で検証する。
- `tests/ClientDispatchTest.java`はHeadless環境のSwingパネルと`GameClient`を使い、受信データのControllerへの振り分け、狭い画面でのTheme／Time表示、共通待機時間後の画面遷移を検証する。
- `./scripts/test.sh`で両テストを連続実行できる。
- WSL2のサーバーをLAN内へ公開する場合は、Windows側で8080番ポートの`portproxy`とファイアウォール受信許可を設定し、クライアントからWindowsのLAN側IPへ接続する。詳しい手順は`RUNNING.md`を参照する。

## 6.9 既知の制約

- ラウンド数はゲーム開始時の参加人数と同じ回数に固定される。ゲーム開始後の途中参加・ルーム移動は拒否され、ゲーム中に参加者が2人未満になると安全に中断される。
- サーバーはポート8080に固定（`server/GameServer.java`のPORT定数）。複数サーバーを同時起動することはできない。

## 6.10 開発時の教訓（同種のバグを再発させないために）

- **`java -cp out ...`はソースを再ビルドしない。** `.class`をそのまま実行するだけなので、修正のたびに必ず先に`javac -d out ...`でコンパイルし直すこと。「サーバーを再起動したのに直っていない」と感じたら、まずビルドし直しているか確認する。
- **`JSplitPane`の初期分割位置は、コンポーネントがまだ画面に表示されていない（サイズが0の）タイミングで`setDividerLocation()`を呼んでも効果がない。** `setResizeWeight()`はリサイズ時の配分比率を決めるだけで初期位置には影響しない。CardLayoutなどで後から表示される画面に分割ペインを置く場合は、実際に表示を切り替えるタイミング（ウィンドウが確実に実サイズを持った後）で`setDividerLocation()`を呼び直すこと。この種のレイアウト不具合は通信ログやコードレベルの統合テストでは再現できず、実機のスクリーンショットで初めて発覚した。見た目に関する不具合の切り分けには、可能な限り実機のスクリーンショットを早めに確認するのが近道。
- サーバー側でプレイヤーを識別する際は、`ClientHandler`オブジェクトそのものではなく文字列のユーザー名をキーに使う設計になっている箇所が多いため、**同名ユーザーが同じ部屋に存在しないことをどこかで保証する**必要がある（現状は`RoomManager.ensureUniqueName()`が担っている）。新しい機能を追加する際、ユーザー名をキーにしたデータ構造を増やす場合は、この前提が崩れていないか意識すること。
